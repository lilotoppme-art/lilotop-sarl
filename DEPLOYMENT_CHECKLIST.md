# LILOTOP SARL - Deployment Checklist v1.0

## Project Audit

- Framework: static HTML site generated by `build-site.js`.
- Runtime: Vercel static hosting with CommonJS Serverless Functions in `api/`.
- Node.js used locally by Codex: `v24.14.0`.
- Next.js: not used.
- TypeScript: not used.
- Build command: `npm run build`.
- Output directory: project root (`.`).

## Public Domain

- Current canonical fallback in code: `https://lilotopsarl.com`.
- Build-time canonical variable: `NEXT_PUBLIC_SITE_URL`.
- Note: the deployment mission mentions `lilotop.com`, but the existing live domain and project content use `lilotopsarl.com`. Do not change the canonical domain without explicit confirmation.

## API Routes

- `POST /api/contact`
  - General contact and contact by department.
  - Reference format: `CNT-YYYYMMDD-XXXX`.
  - Recipient variable: `EMAIL_CONTACT_TO`.

- `POST /api/rfq`
  - Request for quotation / RFQ.
  - Reference format: `RFQ-YYYYMMDD-XXXX`.
  - Recipient variable: `EMAIL_RFQ_TO`.
  - Optional acknowledgement: `RFQ_SEND_ACK=true`.

- `POST /api/portal`
  - Supplier registration.
  - Partnership proposal.
  - Tender / consultation submission.
  - Reference formats: `SUP-YYYYMMDD-XXXX`, `PART-YYYYMMDD-XXXX`, `TND-YYYYMMDD-XXXX`.
  - Recipient variables: `EMAIL_SUPPLIER_TO`, `EMAIL_PARTNERSHIP_TO`, `EMAIL_TENDER_TO`.

## Required Vercel Environment Variables

Configure these for Preview and Production:

```text
RESEND_API_KEY
EMAIL_FROM
EMAIL_REPLY_TO
EMAIL_CONTACT_TO
EMAIL_RFQ_TO
EMAIL_SUPPLIER_TO
EMAIL_PARTNERSHIP_TO
EMAIL_TENDER_TO
EMAIL_TEST_MODE
EMAIL_TEST_TO
RFQ_SEND_ACK
NEXT_PUBLIC_SITE_URL
```

Security:

- `RESEND_API_KEY` must be marked sensitive.
- Do not expose `RESEND_API_KEY` as `NEXT_PUBLIC_*`.
- Use `EMAIL_TEST_MODE=true` and `EMAIL_TEST_TO` in Preview unless real recipients are intentionally tested.
- Use `EMAIL_TEST_MODE=false` in Production.

## Recommended Values To Fill Manually

Do not copy generic examples into production without verification.

- `RESEND_API_KEY`: generated in Resend.
- `EMAIL_FROM`: verified Resend sender, for example `LILOTOP Website <notifications@updates.lilotopsarl.com>` after domain verification.
- `EMAIL_REPLY_TO`: official reply address, currently expected to be `contact@lilotopsarl.com`.
- Department recipients may initially all point to `contact@lilotopsarl.com` until dedicated mailboxes are active.
- `NEXT_PUBLIC_SITE_URL`: canonical public URL, currently expected to be `https://lilotopsarl.com`.

## Resend Domain

- Preferred sending subdomain: `updates.lilotopsarl.com` or `mail.lilotopsarl.com`.
- DNS records must be copied directly from the Resend dashboard.

## Deployment Plan

1. Install dependencies with `npm install`.
2. Configure Resend sending domain.
3. Configure Preview variables in Vercel.
4. Create branch `release/v1.0`.
5. Push branch and generate Vercel Preview.
6. Run real test submissions using:
   - Company: `LILOTOP TEST DEPLOYMENT`
   - Name: `Test Production`
   - Subject: `TEST V1.0 - NE PAS TRAITER`
7. Confirm delivery in Resend and mailbox.
8. Configure Production variables.
9. Merge `release/v1.0` into `main`.
10. Deploy Production.
11. Repeat real tests on production domain.
12. Tag release: `v1.0.0`.

## Rollback Plan

- Keep the previous stable Vercel deployment available.
- Do not delete old deployments.
- If forms fail critically, restore the previous deployment in Vercel.
- If email delivery fails, disable submission paths temporarily or set `EMAIL_TEST_MODE=true` while investigating.
- Review Vercel and Resend logs before redeploying.
- Keep a non-secret list of variables and DNS records for audit.
